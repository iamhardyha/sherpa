Sherpa 🧭

작업에만 몰두하세요. 산출물은 AI가, 신뢰는 코드가.

AI가 만드는 모든 문서를 AI가 끝까지 관리하고, 개발자는 보고받고 승인만 합니다. 그리고 — 반드시 지킬 규칙은 AI에게 부탁하지 않고, 코드가 기계적으로 강제합니다.

v0.2.043 testsNode + TypeScriptgithub.com/iamhardyha/sherpa
근본 명제

신뢰는 프롬프트가 아니라, 프롬프트 밖의 결정론적 강제에서 온다.

LLM의 지시 준수는 확률적입니다. 컨텍스트 압축·핸드오프·새 세션에서 깨집니다. 그래서 Sherpa는 반드시 지킬 불변식을 advisory(권고 텍스트)에서 빼내, hook·CI·정적 파서라는 기계로 옮깁니다. 코드는 잊지 않습니다.

01 — 왜

AI와 오래 일하면
누구나 겪는 세 가지 고통

고통 01

뭐가 살아있지?

결정·계획·명세가 폴더에 쌓인다. 몇 달 뒤 열면 이게 유효한지, 폐기됐는지, 코드랑 맞는지 알 수 없다.

고통 02

맥락이 증발

다른 컴퓨터·새 세션·컨텍스트 압축을 거치면 "내가 어디까지 했지?"가 사라진다. AI도, 당신도.

고통 03

AI는 잊는다

"끝나면 문서 정리해줘"라고 시켜도 LLM은 확률적으로 따른다. 맥락이 끊기는 지점에서 특히.

02 — 철학

다섯 개의 근본 원리

모든 설계가 여기서 나옵니다. 하나씩, 왜 그런지와 함께.

1

결정론적 강제Deterministic Enforcement

LLM에게 규칙을 주는 방법은 둘. 부탁(프롬프트)과 검사(코드). Sherpa는 둘 다 쓰되, 반드시 지킬 소수는 코드로 강제합니다.

방법 A · 부탁 (advisory)
"작업 끝에 문서 정리해줘"
✕ 확률적으로 지켜짐
✕ 압축·핸드오프에서 깨짐
방법 B · 검사 (Sherpa)
hook이 커밋 직전 검사
✓ 항상 발화함
✓ 절대 깨지지 않음

완료된 plan에 report 없이 git commit을 시도하면 — hook이 커밋을 막고 "report가 필요합니다"라고 알립니다. AI가 깜빡해도 코드는 안 깜빡합니다.

2

경량 컨텍스트Lightweight Context · O(1)

AI를 책상에서 일하는 사람으로 보세요. 책장(디스크)엔 문서가 무한히 쌓여도 되지만, 책상(컨텍스트)은 좁습니다. 다 올리면 길을 잃습니다.

📚 책장 — 디스크
adr·plan·report 1년치가 쌓여도 OK. 누적은 O(n).
🪑 책상 — 컨텍스트
좁다. 매번 짊어지는 양은 일정해야. 상시 로드는 O(1).

문서를 통째로 읽지 않고 graph에서 필요한 사실만 질의합니다. 항상 켜둘 규칙은 짧게(≤200줄), 상세는 필요할 때만. 죽은 문서는 묘지로 치웁니다.

3

데이터는 기계가, 그림은 AI가Facts by Machine · Visuals by AI

"이 plan이 끊겼나?", "이 결정이 폐기됐나?" — 이런 사실 판정을 LLM에 맡기면 환각이 생깁니다. 정적 파서가 링크를 읽어 판정하니 지어낼 수가 없습니다.

graph 파서 (기계)

링크를 정적 파싱
= 사실 · 검증 가능
→
lens / 대시보드 (AI)
그 사실을 요약 + HTML로
사실은 못 바꾸고
표현만 더한다
4

시간축 산출물Time-Axis Artifacts

문서 종류의 역할이 겹치면 "이걸 어디다 쓰지?" 혼란이 생깁니다. Sherpa는 산출물을 시간축으로 정렬해 역할이 절대 안 겹치게 합니다. (아래 03 참고.)

왜(adr) → 무엇을 할(plan) → 지금 어떻게(spec) → 무엇을 했(report) → 은퇴(archive). 질문이 다르니 헷갈릴 일이 없습니다.

5

가역성 게이트Reversibility Gate

매 커밋마다 승인을 물으면 곧 읽지도 않고 OK를 누릅니다(고무도장). 그러면 정작 위험한 배포·발행의 주의가 흐려집니다. 그래서 기준은 "중요한가"가 아니라 "되돌릴 수 있는가".

가역 — 자동 진행
로컬 편집 · 로컬 커밋
git으로 되돌림 → 사후 보고
비가역 — 사람 승인
force-push · 릴리스 · 배포
되돌릴 수 없는 8%에만 주의

04 — 동작 원리

강제는 실제로 도는 코드다

설치하면 두 개의 hook이 당신의 작업에 조용히 끼어듭니다. 하나는 비차단, 하나는 차단형.

① 파일 편집 후PostToolUse · 비차단
▸Edit / Write 가 끝나면 자동 발화
↓graph 파서가 docs/를 읽어
↓.graph.json · 대시보드 · INDEX 재생성
✓당신의 "현재 상태"가 항상 최신 — 작업은 막지 않음
② git commit 시도 시PreToolUse · 차단형
▸Bash(git commit) 직전 자동 검사
✕완료된 plan인데 report 없음 → 차단
✕코드 바뀌었는데 spec 드리프트 → 차단
↩무엇이 빠졌는지 알려주고 멈춤 (exit 2)

그 아래에서 graph 파서(43개 테스트로 검증된 TypeScript)가 산출물의 링크를 정적 파싱해 — 끊긴 작업·미처리 결정·드리프트·은퇴 후보를 기계가 판정합니다. LLM이 아니라요.

05 — 근본 원리

왜 이것이 무너지지 않나

표면의 기능이 아니라, 그 아래에서 신뢰를 떠받치는 메커니즘.

경계 밖의 강제Enforcement outside the agent
강제가 에이전트 프롬프트 안이 아니라 git·파일 경계 밖(hook·CI·정적 파서)에 있다. 그래서 누가·몇이 일하든 같은 경계에서 같은 불변식이 발화 — 단일·서브에이전트·하네스·에이전트 팀 무관하게 동일하게 움직인다.
정적 파싱 = 사실Static parsing as truth
역참조 링크를 기계가 읽어 노드·엣지·파생 사실을 산출. 환각이 구조적으로 불가능하다. lens·대시보드는 이 사실을 소비할 뿐, 새로 판정하지 않는다.
누적 ≠ 상시 로드Accumulation ≠ working set
디스크는 무한히 쌓여도(O(n)), 컨텍스트에 들어오는 건 질의로 묶인 슬라이스(O(1)). 비대해진 컨텍스트가 곧 길 잃음이므로, 경량화는 보너스가 아니라 전제다.
가역성 = 주의 배분Reversibility as attention
비가역 행위는 전체의 ~0.8%. 가역 행위를 게이트하지 않음으로써, 정작 위험한 소수에 사람의 주의를 모은다. 고무도장을 구조적으로 막는다.
트리비얼 면제Trivial exemption
사소한 변경엔 비싼 판단층(리뷰·테스트)을 면제하되, 값싼 결정론 바닥(링크·드리프트 검사)은 항상 실행. 면제가 안전한 전제가 그 바닥이다. 판정은 LLM이 아니라 기계가.

언제 무엇을 쓰나

/sherpa-init
프로젝트 시작 시 1회 — .sherpa·docs 구조를 결정론적으로 생성

"내가 어디 있지?" — 진행 중 작업·주의 항목을 대시보드로

전체 그림이 보고 싶을 때 — 요약 + 시각 HTML로 조망

코드 규칙을 정할 때 — 팀/개인 컨벤션을 린터로 매핑
/sherpa-sweep
죽은 문서 정리 — 은퇴 후보 제안, 승인분만 archive
/sherpa-reindex
수동 재생성 (보통 hook이 자동) — INDEX·대시보드·graph
