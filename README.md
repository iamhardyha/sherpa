# Sherpa 🧭

> **작업에만 몰두하세요. 산출물 관리는 AI가, 신뢰는 코드가 책임집니다.**

### 📊 [**라이브 소개 페이지 보기 →**](https://iamhardyha.github.io/sherpa/)
*철학 · 산출물 · 동작 원리 · 근본 원리를 시각 자료로*

Sherpa는 Claude Code로 개발할 때 **길을 잃지 않게** 해주는 플러그인입니다.
AI가 만들어내는 모든 문서 — 결정·계획·명세·보고 — 를 AI가 끝까지 관리하고,
개발자는 **보고받고 승인**하는 데만 집중합니다.

그리고 가장 중요한 차이: "반드시 지켜야 하는 규칙"을 AI에게 *부탁*하지 않습니다.
**코드가 기계적으로 강제**합니다.

---

## 30초 요약

```
당신이 하는 일          Sherpa가 하는 일
─────────────         ──────────────────────────────────
코드를 짠다       →    무슨 결정·계획·명세인지 문서로 남긴다 (adr·plan·spec·report)
"이거 해줘"       →    plan을 세우고, 작업하고, 검증하고, 보고한다
승인한다          →    되돌릴 수 없는 일(배포·발행)만 당신에게 묻는다
길을 잃는다       →    /sherpa-status · /sherpa-lens 로 "지금 어디" 를 보여준다
문서를 깜빡한다   →    hook이 커밋 직전에 막는다 ("이 작업, report가 없는데요?")
```

핵심은 마지막 줄입니다. Sherpa는 **AI의 성실함에 기대지 않습니다.** 코드가 검사합니다.

---

## 왜 필요한가요?

AI와 오래 일하다 보면 누구나 겪는 세 가지 고통이 있습니다.

**1. 문서가 쌓이는데, 뭐가 살아있는지 모른다.**
결정(ADR)·계획(plan)·명세(spec)가 폴더에 쌓입니다. 그런데 몇 달 뒤 열어보면
*이게 아직 유효한 결정인지, 폐기된 건지, 코드랑 맞는지* 알 수가 없습니다.

**2. 맥락이 증발한다.**
다른 컴퓨터로 옮기거나, 새 세션을 열거나, 컨텍스트가 압축되면
**"내가 어디까지 했지?"** 가 사라집니다. AI도, 당신도.

**3. AI는 잊는다 — 그것도 확률적으로.**
"작업 끝나면 문서 정리해줘"라고 아무리 시켜도, LLM은 **확률적으로** 따릅니다.
특히 핸드오프·압축처럼 맥락이 끊기는 지점에서 잘 잊습니다.
프롬프트로 준 규칙은 *대체로* 지켜지지만, *항상*은 아닙니다.

Sherpa의 답은 한 문장입니다:

> ### 신뢰는 프롬프트가 아니라, 프롬프트 *밖*의 결정론적 강제에서 온다.

AI에게 "잊지 마"라고 부탁하는 대신, **hook과 정적 파서가 기계적으로** 검사합니다.
코드는 압축되지 않고, 핸드오프에서 깨지지 않고, 기분에 따라 변하지 않습니다.

## 핵심 철학

Sherpa의 모든 설계는 다섯 개의 원칙에서 나옵니다. 하나씩, *왜* 그런지와 함께.

### 1. 결정론적 강제 — "부탁"이 아니라 "검사"

LLM에게 규칙을 주는 방법은 두 가지입니다.

```
방법 A (대부분의 도구)         방법 B (Sherpa)
─────────────────────         ──────────────────────────
"작업 끝에 문서 정리해줘"  →   hook이 커밋 직전에 검사한다
= 프롬프트 (advisory)          = 코드 (deterministic)
= 확률적으로 지켜짐            = 항상 발화함
= 압축·핸드오프에서 깨짐       = 깨지지 않음
```

Sherpa는 **둘 다** 씁니다. 부드러운 안내는 프롬프트(스킬)로, **반드시 지킬 소수는 코드로**.
예를 들어 — 태스크를 전부 끝낸 plan에 report 없이 `git commit`을 시도하면,
PreToolUse hook이 **커밋을 막고** "이 작업, report가 필요합니다"라고 알려줍니다.
AI가 깜빡해도 코드는 안 깜빡합니다.

> 💡 이 철학은 Sherpa만의 발명이 아닙니다. 2026년 Claude Code 생태계의 떠오르는
> 베스트 프랙티스 — *"instructions에 기대지 말고 hook으로 가드레일을 강제하라"* — 와 같은 방향입니다.
> Sherpa는 그걸 **재사용 가능한 플러그인**으로 묶었습니다.

### 2. 경량 컨텍스트 — 책장은 무한, 책상은 일정

AI를 책상에서 일하는 사람이라고 생각해보세요.

```
📚 책장 (디스크)              🪑 책상 (AI 컨텍스트)
문서 1년치가 쌓여도 OK        좁다 — 몇 권만 올라간다
adr·plan·report 무한 누적     다 올리면 일을 못 하고 길을 잃는다
```

문제는, 프로젝트가 커지면 문서(책)가 계속 늘어난다는 겁니다.
일할 때마다 그걸 전부 책상에 올리면 → AI가 헷갈리고 느려지고 정확도가 떨어집니다.

그래서 Sherpa의 불변식:

> **누적은 O(n)이어도, 상시 로드는 O(1).**
> 문서가 아무리 쌓여도 AI가 *매번 짊어지는* 맥락의 양은 일정해야 한다.

방법은 셋입니다 — ① 항상 켜두는 규칙은 짧게(≤200줄), 상세는 필요할 때만 꺼내는
스킬로(progressive disclosure), ② 문서를 통째로 읽는 대신 **graph에서 필요한 사실만 질의**,
③ 죽은 문서는 archive(묘지)로 치워 책상에서 뺀다.

### 3. 데이터는 기계가, 그림은 AI가 — 환각을 구조적으로 차단

"이 plan이 끊겼나?", "이 결정이 폐기됐나?", "이 spec이 코드랑 안 맞나?"
— 이런 **사실 판정을 LLM에게 맡기면 환각이 생깁니다.**

Sherpa는 이 사실들을 **정적 파서(graph 파서)가 결정론적으로** 산출합니다.
링크를 기계가 읽어 판정하니 *지어낼 수가 없습니다.*
AI는 그 사실 **위에** 요약과 시각화만 얹습니다.

```
graph 파서 (기계)              lens / 대시보드 (AI)
─────────────────              ──────────────────────
"plan 0012는 끊겼다"     →     그 사실을 예쁜 HTML로, 요약 문단과 함께
"ADR 0017은 폐기됐다"          (사실은 못 바꾸고, 표현만 더한다)
= 사실 (검증 가능)             = 그림 (이해를 돕는)
```

### 4. 시간축 산출물 — "어디다 쓰지?"를 없앤다

문서 종류의 역할이 겹치면 *"이걸 ADR에 쓰나, plan에 쓰나?"* 혼란이 생깁니다.
Sherpa는 산출물을 **시간축**으로 정렬해 역할이 절대 안 겹치게 합니다.

```
   결정할 때      작업 전        상시         작업 후       은퇴
   ─────────     ─────────     ─────────    ─────────    ─────────
     adr     →     plan    →     spec    →    report   →   archive
     왜?          무엇을 할?     지금 어떻게?   무엇을 했?    더는 현역 아님
```

질문이 다르니 헷갈릴 일이 없습니다. *왜* 정했는지는 adr, *무엇을 할지*는 plan,
*지금 어떻게 도는지*는 spec, *실제로 무엇을 했는지*는 report.

### 5. 가역성 게이트 — 고무도장을 막는다

매 커밋마다 "승인하시겠어요?"를 물으면, 당신은 곧 **읽지도 않고 OK를 누르게** 됩니다(고무도장).
그러면 정작 위험한 행위 — 배포·릴리스·외부 발행 — 의 주의가 흐려집니다.

그래서 Sherpa의 승인 기준은 *"중요한가"*가 아니라 ***"되돌릴 수 있는가"*** 입니다.

```
가역 (git으로 되돌림)           비가역 (되돌릴 수 없음)
로컬 편집 · 로컬 커밋           force-push · 릴리스 태그 · 배포 · 외부 발행
→ 자동으로 진행, 사후 보고      → 반드시 사람이 명시 승인
```

되돌릴 수 있는 일에는 길을 비켜주고, **되돌릴 수 없는 8%에만** 당신의 주의를 모읍니다.

## 설치

```bash
# 1. 마켓플레이스 추가
/plugin marketplace add iamhardyha/sherpa

# 2. 플러그인 설치
/plugin install sherpa@sherpa-marketplace

# 3. 현재 세션에 활성화
```

설치하면 hook(편집 후 graph 재생성 · 커밋 전 트리거 검사)이 **자동으로 활성화**됩니다.
별도 설정은 필요 없습니다.

**요구사항**: [Claude Code](https://claude.com/claude-code) · Node.js
(Claude Code가 이미 Node.js로 동작하므로 **추가 설치가 필요 없습니다**).

## 빠른 시작

```bash
# 프로젝트를 Sherpa 구조로 1회 초기화
/sherpa-init
```

이러면 `.sherpa/`(거버넌스)와 `docs/`(산출물 폴더)가 생깁니다. 그 다음엔 — **그냥 일하면 됩니다.**

```
당신:  "사용자 인증 기능 추가해줘"

Sherpa가 자동으로:
  1. plan 생성        docs/plan/  에 태스크 체크박스로 계획
  2. 작업 + 테스트     코드를 짜고 테스트 작성
  3. 산출물 갱신       spec 갱신 · report 작성
  4. graph 재생성      편집할 때마다 hook이 .graph.json·대시보드 갱신 (자동)
  5. 검증 · 보고       /simplify → /code-review, 결과 보고
  6. 커밋              가역이라 자동 — 단, report 빠지면 hook이 막음

길을 잃었다 싶으면:
  /sherpa-status   "지금 진행 중인 작업과 주의 항목"
  /sherpa-lens     "전체를 한눈에" (요약 + 시각 HTML)
```

## 산출물 — Sherpa가 관리하는 6종

다섯 종은 시간축 위에 있고(서로 안 겹침), 한 종(lens)은 그 전체를 가로질러 봅니다.

| 산출물 | 묻는 질문 | 핵심 성질 |
|---|---|---|
| **adr** | *왜* 이렇게 정했나 | 결정 1건 = 1장, append-only. 번복은 새 ADR + 양방향 supersede 링크 |
| **plan** | 앞으로 *무엇을* 할 건가 | `## 작업`의 태스크 체크박스로 진척도 추적, 생명주기 상태(active·done·…) |
| **spec** | 지금 *어떻게* 동작하나 | 살아있는 단일 진실. `refs`로 코드와 자동 대조 → 어긋나면 드리프트 감지 |
| **report** | 실제로 *무엇을* 했나 | plan을 역참조 → "이 계획이 어떻게 끝났나"가 한 줄기로 추적됨 |
| **archive** | 더는 현역 아님 | 은퇴한 산출물의 묘지 (사람이 승인한 것만 이동) |
| 🔭 **lens** | 전체를 *한눈에* | docs를 읽어 **요약 + 시각 HTML**로 조망. "길 잃었을 때 들여다보는" 도구 |

### 폴더는 이렇게 생겼습니다

```
your-project/
├── .sherpa/                  ← 거버넌스 (AI가 항상 참조, git에 커밋)
│   ├── workflow.md           일하는 법 (핵심만 — 상세는 스킬로)
│   ├── project-context.md    이 레포가 무엇인가
│   ├── conventions.md        아키텍처 + 코드 규칙
│   └── progress.md           AI 작업 메모리 (세션 간 복원용)
│
└── docs/
    ├── adr/      plan/      spec/      report/     ← 산출물 + 각 INDEX
    ├── lens/                                       ← 조망 HTML (뷰별)
    ├── archive/                                    ← 묘지 (은퇴)
    ├── README.md                                   ← 대시보드 (자동 생성)
    └── .graph.json                                 ← 기계가 읽는 사실 (자동, gitignore)
```

> `.graph.json`·`README.md`·`INDEX.md`는 **손대지 마세요.** 폴더에서 자동 재생성됩니다.

## 명령 — 언제 무엇을 쓰나

| 명령 | 언제 쓰나 |
|---|---|
| `/sherpa-init` | **프로젝트 시작 시 1회.** `.sherpa/`·`docs/` 구조를 결정론적으로 생성 |
| `/sherpa-status` | **"내가 어디 있지?"** 궁금할 때. 진행 중 작업·주의 항목을 대시보드로 |
| `/sherpa-lens` | **전체 그림이 보고 싶을 때.** "무엇을 들여다볼까요?" → 요약 + 시각 HTML |
| `/sherpa-conventions` | **코드 규칙을 정할 때.** 팀/개인 컨벤션을 캡처하고 가능한 건 린터로 매핑 |
| `/sherpa-sweep` | **죽은 문서를 정리할 때.** 은퇴 후보를 제안 → 당신이 승인한 것만 archive |
| `/sherpa-reindex` | **수동 재생성** (보통은 hook이 자동으로 함). INDEX·대시보드·graph 재생성 |

## 결정론 바닥이 실제로 도는 법

Sherpa의 "강제"는 추상적 약속이 아니라 **실제로 도는 코드**입니다.
설치하면 두 개의 hook이 당신의 작업에 조용히 끼어듭니다.

```
① 파일을 편집한다 (Edit / Write)
        │
        ▼  PostToolUse hook  (비차단 — 작업을 막지 않음)
   graph 파서가 docs/를 읽어
   .graph.json + 대시보드 + INDEX 재생성
   → 당신의 "현재 상태"가 항상 최신


② git commit 을 시도한다
        │
        ▼  PreToolUse hook  (차단형 — 미충족이면 커밋을 멈춤)
   산출물 트리거 검사:
     • 완료된 plan인데 report가 없다     → 🚫 차단
     • 코드가 바뀌었는데 spec이 드리프트  → 🚫 차단
   → 충족되면 통과, 아니면 "무엇이 빠졌는지" 알려줌
```

그 아래에서 **graph 파서**(43개 테스트로 검증된 TypeScript)가 일합니다 —
산출물의 링크를 정적 파싱해 노드·엣지·파생 사실(끊긴 작업·미처리 결정·드리프트·은퇴 후보)을
**기계가 판정**합니다. LLM이 아니라요.

## 실행 방식과 무관하게 동일합니다

단일 에이전트로 일하든, 서브에이전트를 쓰든, 하네스나 **에이전트 팀**으로 일하든 —
Sherpa는 **똑같이 움직입니다.**

이유는 간단합니다. 강제가 *에이전트의 프롬프트 안*이 아니라
**git·파일 경계 밖(hook·CI·정적 파서)**에 있기 때문입니다.
누가, 몇이 작업하든 같은 경계에서 같은 불변식이 발화합니다.
서로 다른 에이전트가 advisory를 못 물려받아도, hook은 항상 거기 있습니다.

## 한눈에 보는 설계 원칙

| 원칙 | 한 줄 |
|---|---|
| 🧱 결정론적 강제 | 반드시 지킬 건 프롬프트가 아니라 hook/CI가 막는다 |
| 🪶 경량 컨텍스트 | 문서가 쌓여도 AI가 짊어지는 맥락은 일정 (O(1)) |
| 🤖 데이터는 기계가 | 사실은 파서가 판정(환각 불가), AI는 요약·시각화만 |
| ⏳ 시간축 산출물 | 왜→무엇을할→지금어떻게→무엇을했→은퇴 (역할 안 겹침) |
| ↩️ 가역성 게이트 | 되돌릴 수 있으면 자동, 비가역만 사람 승인 |

## 현재 상태
