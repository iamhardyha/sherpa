# Sherpa 🧭

> **작업에만 몰두하세요. 산출물 관리는 AI가, 신뢰는 코드가 책임집니다.**

Sherpa는 Claude Code로 개발할 때 **길을 잃지 않게** 해주는 플러그인입니다. AI가 만들어내는 모든 문서(결정·계획·명세·보고)를 AI가 끝까지 관리하고, 개발자는 **보고받고 승인**하는 데만 집중합니다.

---

## 왜 필요한가요?

AI와 오래 일하다 보면 누구나 겪는 일입니다.

- 📄 결정·계획·명세 문서가 여기저기 쌓이는데, **뭐가 최신이고 뭐가 죽었는지** 알 수 없습니다.
- 🔀 다른 컴퓨터·새 세션·컨텍스트 압축을 거치면 **"내가 어디까지 했지?"** 가 사라집니다.
- 🤖 "작업 끝에 문서 정리해줘"라고 아무리 시켜도, LLM은 **확률적으로** 까먹습니다. 핸드오프·압축 지점에서 특히.

Sherpa의 답은 하나입니다:

> **신뢰는 프롬프트가 아니라 프롬프트 밖의 결정론적 강제(hook·CI·정적 파서)에서 온다.**

"꼭 지켜야 하는 규칙"은 AI에게 부탁하지 않습니다. **코드가 기계적으로 강제**합니다.

---

## 핵심 아이디어 3가지

| | 아이디어 | 무슨 뜻인가요 |
|---|---|---|
| 🧱 | **결정론적 강제** | 끊긴 작업·드리프트·미처리 결정을 LLM이 "판단"하지 않습니다. 정적 파서가 링크를 읽어 **기계가 판정**합니다. AI는 그 사실 위에 요약·시각화만. |
| 🪶 | **경량 컨텍스트** | 문서가 1년치 쌓여도 AI가 매번 짊어지는 맥락은 **항상 일정**합니다 *(누적은 O(n), 상시 로드는 O(1))*. 전부 읽지 않고 필요한 사실만 질의합니다. |
| ⏳ | **시간축 산출물** | 산출물 5종이 시간축으로 정렬돼 역할이 겹치지 않습니다 — *왜(adr) · 무엇을 할(plan) · 지금 어떻게(spec) · 무엇을 했(report) · 은퇴(archive)*. "어디에 뭘 쓰지?" 고민이 사라집니다. |

---

## 산출물 한눈에

| 산출물 | 질문 | 핵심 |
|---|---|---|
| **adr** | *왜* 정했나 | 결정 1건 1장, append-only, 양방향 supersede 링크 |
| **plan** | 앞으로 *무엇을* 할 건가 | 태스크 체크박스로 진척도 추적, 생명주기 상태 |
| **spec** | 지금 *어떻게* 동작하나 | 살아있는 단일 진실, 코드와 자동 대조(드리프트 감지) |
| **report** | 실제로 *무엇을* 했나 | plan을 역참조 → "이 계획이 어떻게 끝났나" 추적 |
| **archive** | 더는 현역 아님 | 묘지로 정리(승인받은 것만) |
| 🔭 **lens** | 전체를 *한눈에* | docs 전체를 읽어 **요약 + 시각 HTML**로 조망 |

## 명령

작업 흐름에 맞춰 6개의 명령을 제공합니다.

| 명령 | 하는 일 |
|---|---|
| `/sherpa-init` | 레포를 Sherpa 구조로 1회 초기화 |
| `/sherpa-conventions` | 팀/개인 컨벤션(아키텍처·코드 규칙)을 캡처하고 린터에 매핑 |
| `/sherpa-status` | "내가 어디 있나" — 진행 중 작업·주의 항목을 대시보드로 |
| `/sherpa-lens` | "어떤 걸 들여다볼까요?" → 요약 + 시각 HTML로 조망 |
| `/sherpa-sweep` | 은퇴한 산출물을 archive 후보로 제안 (승인분만 이동) |
| `/sherpa-reindex` | INDEX·대시보드·그래프를 폴더에서 재생성 |

## 워크플로우

실행 방식(단일 에이전트·서브에이전트·하네스·에이전트 팀)과 **무관하게 동일하게** 움직입니다. 강제가 git/파일 경계 밖(hook·CI)에 있기 때문입니다.

```
1. plan      트리비얼이 아니면 계획부터
2. 작업      수행
3. 테스트    (로직이면) 단위·통합·인수
4. 산출물    spec 갱신 · report 작성 · archive 정리
5. 검증      문서=셀프검증 / 로직=/simplify → /code-review + 테스트
6. 보고
7. 게이트    되돌릴 수 있으면 자동, 비가역(배포·발행)만 사람이 승인
```

승인은 **"중요한가"가 아니라 "되돌릴 수 있는가"** 기준입니다. 매 커밋마다 승인을 받으면 고무도장이 되어, 정작 위험한 행위(배포·릴리스)의 주의가 흐려지니까요.

---

## 철학

- **데이터는 기계가, 그림은 AI가.** 사실 판정은 결정론적 파서가, 요약·시각화는 LLM이. 환각이 구조적으로 불가능합니다.
- **가벼움이 곧 신뢰.** 컨텍스트가 비대해지면 AI는 길을 잃습니다. 그래서 경량화는 보너스가 아니라 전제입니다.
- **강제는 보너스가 아니라 바닥.** advisory(권고 텍스트)는 깨질 수 있어도, hook/CI는 항상 발화합니다.

## 현재 상태

> ⚠️ Sherpa는 **활발히 개발 중**입니다. 결정론 바닥(코드 강제)이 동작하며, 일부 명령을 마저 잇고 있습니다.

- ✅ 설계 스펙 확정 · 산출물/거버넌스 템플릿 · 스킬(core-workflow·lens·conventions) · 명령 정의
- ✅ **graph 파서** — 노드·엣지·파생 사실(끊긴 작업·미처리 결정·드리프트) 판정 + `.graph.json` 생성 (43 tests)
- ✅ **`/sherpa-init`** — 결정론적 스캐폴드(거버넌스 + docs 구조)
- ✅ **결정론 강제 hook** — PostToolUse(편집 후 graph 재생성, 비차단) · PreToolUse(커밋 전 산출물 트리거 검사, 차단)
- ✅ 링크 체커 · 트리비얼 분류기(§6-D)
- 🔨 나머지 명령 동작(`status`·`sweep`·`lens`) · lens HTML 생성 — 진행 중
