# core-workflow — 불변 워크플로우 상세 절차

거버넌스 `.sherpa/workflow.md`는 짧은 핵심만 담는다(§4.1 progressive disclosure). 그 상세 절차·근거·예외가 여기 있다. **정본 규칙은 설계 스펙 §6**(A~K) — 이 스킬은 *어떻게 실행하나*를 안내하고 스펙으로 링크백한다.

## 대전제 — 신뢰는 프롬프트가 아니라 결정론 바닥에서 (§4.3)

LLM의 지시 준수는 확률적이라 압축·핸드오프에서 깨진다. 그래서 *반드시 지킬 불변식*은 이 텍스트가 아니라 **hook/CI/정적 파서**가 강제한다. 이 스킬을 읽었든 안 읽었든 바닥은 발화한다 — 스킬은 advisory(어떻게 잘 하나), 바닥은 강제(반드시 통과).

## 불변 작업 순서 (§6-C — 실행 모드 무관: 단일·서브에이전트·하네스·팀)

```
1. plan      트리비얼 아니면 생성 (설계 필요면 브레인스토밍)
2. 작업      수행
3. 테스트    (로직) 단위·통합·인수
4. 트리거    산출물 5종 적용 (spec·report·archive)  → references/artifact-triggers.md
5. 검증      문서=셀프검증 6항목 / 로직=/simplify → /code-review low + 테스트
6. 보고
7. 게이트    가역=자동(HOTL) / 비가역만 사람 승인(HITL)
```

### 1. plan (§2.2·§6-C)

- **트리비얼하면 생략** — 가벼운 단일 변경은 plan 없이. 다중 단계면 plan 생성, `## 작업`을 태스크 체크박스로 세분.
- **브레인스토밍은 기본 OFF.** '설계 필요'일 때만 또는 명시 요청 시 → plan에 `## 설계` 섹션.
  - '설계 필요' 기준: ① 새 기능·서브시스템 ② 아키텍처 영향(여러 모듈·레이어·공유 primitive·공개 API·데이터 모델) ③ 비가역·대안 있는 결정 ④ 요구사항 모호.
  - **경계가 모호하면 추측 말고 물어라** — "설계 브레인스토밍 할까요?"(값싼 escalation).
- 비가역·대안 결정이 있으면 **ADR도**(§2.3). plan 변경이력이 정본, ADR은 링크로(트리플 쓰기 금지 §2.2).
- 식별자 = `{date}-{slug}`(영구), NNNN은 표시용. 역참조는 slug로(§2.8).

### 2~3. 작업 + 테스트

- 코드를 쓸 때 `conventions.md`(아키텍처+코드 규칙)를 따른다(§6-J).
- 로직이면 유형별 테스트: 단위(도메인·UseCase)·통합(어댑터·DB·외부)·인수(컨트롤러~E2E). 사소 변경은 면제(§6-D).

### 4. 산출물 트리거 (§6-A)

작업 끝에 §2 5종 표를 훑되 **이것에만 의존하지 않는다** — 커밋 전 hook이 결정론으로 교차검증. 트리거는 두 갈래로 정확히 정의:
- **spec 트리거 = 드리프트**: 코드가 spec의 `refs`를 바꿨는데 spec 미갱신이면 셀프검증 실패(§6-I, 하드 게이트).
- **report 트리거 = plan 완료**: plan 태스크 전부 체크 + report 없음을 graph가 파생(§2.2·§4.7). *코드 변경 자체가 아니라 계획 종결*이 신호.
- **trivial 변경은 트리거 의무 면제**(§6-D) — 로그 한 줄에 report 강요 금지. 단 드리프트·링크·placeholder 바닥은 그대로.
- 언제 무엇을 만드나의 결정 가이드 → **references/artifact-triggers.md**.

### 5. 검증 (에이전트가 직접 실행 — 지시 대기 없음)

- **문서**: 셀프검증 6항목(①placeholder ②링크 ③코드대조 ④내부일관성 ⑤드리프트 ⑥트리거 교차). ①②⑤⑥은 스크립트가 기계화(§4.3).
- **로직 2단계**(순서가 핵심 §6-E):
  1. **`/simplify`** — 품질 정리(재사용·단순화·효율·고도). 작업 트리에 직접 적용. *버그는 안 봄.*
  2. **`/code-review low`** — 버그·정확성(위험·복잡=medium). 무인자 `/code-review` 금지(high로 튐).
  - simplify가 코드를 바꾸므로 *그 뒤*에 리뷰·테스트가 정리된 최종본을 본다.
  - high/max/ultra는 사람만. 역할 분리: simplify=품질, code-review=버그.
- **합격선**(§6-K): 불변 게이트(테스트 green·code-review CRITICAL/HIGH 해소·드리프트 0·트리거 처리)는 강제. 임계값(커버리지·MEDIUM 수위)은 conventions.md.

### 6~7. 보고 + 가역성 게이트 (§6-F)

기준은 "중요한가"가 아니라 **"되돌릴 수 있는가"**(비가역은 행동의 ~0.8%):
- **자동(HOTL)**: 로컬 편집·로컬 커밋 — git으로 가역. 게이트 없이 진행, report로 사후 노출.
- **체크포인트**: main 머지·PR 오픈·스테이징 배포 → 1회 확인.
- **명시 게이트(HITL)**: force-push·릴리스 태그·외부 발행·프로덕션 마이그레이션 등 **비가역만**. 보고 → (변경 시 재승인) → 명시 승인(답변≠승인).
- 매 커밋 승인은 고무도장을 낳아 정작 위험한 8%의 주의를 흐린다 → 가역 행위는 게이트하지 않는다.

## 트리비얼 면제 — 판단층만, 바닥은 항상 (§6-D)

사소한 변경에 비싼 판단층(`/simplify`·`/code-review`·테스트 작성·내부 일관성 판단)을 돌리면 경량 컨텍스트(§1)에 역행 → **판단층 면제**, 단 **결정론 바닥(링크·placeholder·드리프트)은 trivial이라도 항상**. 면제 판정은 LLM이 아니라 기계가.

- **코드 면제**: diff가 주석/공백·포맷/로그 문구/자명한 리네임뿐임을 기계 확인할 때만. **버그·동작·계약 변경은 1~2줄이라도 면제 아님.** 함정: ①공백은 언어 인지 필수(Python/YAML/탭은 공백이 동작) ②로그는 순수 리터럴/변수일 때만(부작용 호출이면 면제 불가) ③리네임은 리뷰만 면제(`refs`·공개 API면 드리프트가 잡음).
- **문서 면제**: prose 본문 줄만 바뀐 게 기계 확인되면 판단 항목(③④⑥)만 생략, 바닥(①②⑤)은 항상. frontmatter·링크 URL·코드펜스·anchor 건드리면 trivial 아님.

## 세션 복원 (§2.5)

새 세션·compaction 후 **가장 먼저 `.sherpa/progress.md`를 읽어** 상태 복원("지금 plan 0012의 태스크 2"). 완료되면 report로 승격하고 progress는 비운다. 팀/병렬은 `progress/{agent}.md`(§4.6).

## 손대지 말 것

- INDEX·대시보드·`.graph.json` — `/sherpa-reindex`가 재생성(§6-G). 손편집 금지.
- `.sherpa/workflow.md` — `/sherpa-init --upgrade`로만 재생성.
- lens 그래프 사실 — graph 파서가 판정(§4.7). lens는 그 위에 시각화만(§2.4).
